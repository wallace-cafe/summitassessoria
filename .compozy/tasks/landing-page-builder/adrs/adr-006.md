# ADR-006: Static File Serving via PHP Controller Endpoint

## Status

Accepted

## Date

2026-05-20

## Context

Uploaded landing page files (style.css, app.js, images) are stored in `writable/landing_pages/{slug}/` — outside the webroot for security. Visitors' browsers need to load these files when viewing the public page. Three approaches were considered for serving them.

## Decision

Serve all static files through a single PHP controller endpoint at `/p/{slug}/assets/{filepath}`. The controller reads the requested file from the page's filesystem directory and outputs it with the correct MIME type and caching headers.

## Alternatives Considered

### Alternative 1: Copy/Symlink to Public Directory

- **Description**: On upload, copy or symlink static files to `public/assets/landing-pages/{slug}/`. Browsers access them directly as static files.
- **Pros**: Fastest serving (no PHP overhead), standard web server caching.
- **Cons**: Requires sync logic on upload and cleanup on delete; copies/symlinks add deployment complexity; symlinks may not work on all hosting environments.
- **Why rejected**: Adds deployment complexity and sync state management. The security benefit of storing outside webroot is partially undermined.

### Alternative 2: Embed Everything in HTML

- **Description**: Inline CSS in `<style>` tags, JS in `<script>` tags, images as base64 data URIs within the HTML served by PublicController.
- **Pros**: Single request for everything, no extra file serving needed.
- **Cons**: Bloated HTML for image-heavy pages, no browser caching for static assets, base64 is ~33% larger than binary.
- **Why rejected**: Practical pages with images would produce unreasonably large HTML responses.

## Consequences

### Positive

- Files remain outside webroot — no direct filesystem access by visitors
- Full control over MIME types, caching headers, and access control
- No symlink or copy management
- Simple to implement with CodeIgniter's Response object

### Negative

- Every static asset request hits PHP (marginal performance cost for typical landing pages with 3-10 assets)

### Risks

- **Performance under load**: A landing page with many images could generate many PHP requests. Mitigation: set far-future `Cache-Control` headers so browsers cache assets after the first load. If needed later, a reverse proxy (nginx/Varnish) can cache these responses.
- **Large file serving**: PHP reading and outputting large files consumes memory. Mitigation: use `$response->setFilePath()` or read in chunks with `fread()` for files over 2MB.

## Implementation Notes

- Route pattern: `GET /p/{slug}/assets/(:any)` → `PublicController::asset/$1/$2`
- The controller locates the page by slug, resolves the file path under `writable/landing_pages/{slug}/`, validates it exists and is within the allowed directory, and serves it with appropriate Content-Type and Cache-Control headers.
- Path traversal attacks must be prevented by resolving the requested path and verifying it stays within the page's directory.
