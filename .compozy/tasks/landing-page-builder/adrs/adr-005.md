# ADR-005: File Upload Landing Page with Comment-Delimited Block Editing

## Status

Accepted

## Date

2026-05-20

## Context

The existing Landing Page Builder uses a block template library + token substitution system. Users create reusable HTML blocks with `%TOKEN%` placeholders, compose pages by picking and ordering blocks, filling token values, and saving as JSON. This approach was designed for law firm administrators who needed to build pages without coding.

A new requirement has emerged: external designers and agencies produce complete landing pages (HTML, CSS, JS, images) and deliver them to law firm administrators for publishing. The block/token system is too rigid for this workflow — designers need full HTML/CSS/JS freedom without template constraints.

The refactoring must:
- Support uploading a complete landing page package (index.html, style.css, app.js, images)
- Allow administrators to edit text content of delimited sections within the HTML
- Continue providing lead capture on public pages
- Replace the existing block system entirely

## Decision

Adopt **Approach A: Directory-per-page with HTML block parsing**.

Uploaded files are stored in `writable/landing_pages/{slug}/` preserving the original directory structure and relative paths. The `landing_pages` table stores metadata (title, slug) plus a reference to the filesystem path. The public controller reads `index.html` from disk, injects the lead capture form via a `<!-- LEAD_FORM -->` placeholder, and serves the assembled page.

For editing, the system parses `<!-- BLOCO_N_INICIO -->...<!-- BLOCO_N_FIM -->` comment delimiters in `index.html`, extracts the inner text content of each block, and presents a simple text editor per block. Saving rewrites the block's content in the `index.html` file on disk.

## Alternatives Considered

### Alternative 1: Self-contained HTML in DB

- **Description**: Store the entire index.html in a new `html_content` TEXT column. Images must be external URLs. CSS and JS are embedded in the HTML.
- **Pros**: Simple backup/restore (everything in DB), no filesystem cleanup issues, no file permission problems.
- **Cons**: No image upload support, large HTML in DB rows, requires PHP processing for every page view, no support for separate JS/CSS files.
- **Why rejected**: Does not support images or external JS/CSS — defeats the purpose of full creative freedom for designers.

### Alternative 2: Filesystem storage with draft/publish workflow

- **Description**: Same file storage as Approach A but adds a draft copy mechanism. Edits go to a draft; the public URL only updates on explicit "Publish".
- **Pros**: Edit safety — saves are not immediately visible to visitors.
- **Cons**: Added complexity of managing draft vs published files, contradicts the user's stated preference that saves go live immediately.
- **Why rejected**: Over-engineered for MVP; user explicitly chose instant publish.

## Consequences

### Positive

- Full HTML/CSS/JS freedom for external designers and agencies
- Image files work with relative paths (no external CDN dependency)
- File serving can be fast (static files from disk)
- Simple text editing for administrators — no need to understand HTML
- Lead capture is guaranteed on every page via automatic injection
- Clean break from the old block system removes code complexity

### Negative

- File storage introduces operational concerns: backups, permissions, cleanup on page deletion
- Loses the live visual preview from the old builder (previewing requires actually serving the file)
- `writable/uploads/` files are not automatically backed up with the database
- Need to handle file name collisions, directory creation, and cleanup

### Risks

- **File cleanup on page deletion**: If an admin deletes a landing page, the files on disk must be removed. Mitigation: delete the directory recursively in the controller's `delete()` method.
- **Security of uploaded files**: Uploaded HTML/JS files could contain malicious scripts. Mitigation: files are served from a controlled PHP endpoint (not direct filesystem access), and only authenticated users can upload.
- **Large uploads**: A ZIP with many images could exceed PHP upload limits. Mitigation: set reasonable file count/size limits and communicate them in the UI.
- **Malformed block delimiters**: If the uploaded HTML lacks `<!-- BLOCO_N_INICIO/FIM -->` comments, the editor shows no editable blocks. Mitigation: show a warning and fall back to a raw HTML editor.

## Implementation Notes

- Store files in `writable/landing_pages/{slug}/` — outside the webroot for security
- Serve files via a dedicated `LandingPageFileController` or extended `PublicController`
- `landing_pages` table needs a migration: add `file_path` (VARCHAR) column, keep `slug` and `title`, drop `blocks` and `custom_css` columns
- The `block_templates` table and `BlockTemplatesController` can be deprecated/removed
- The `page-builder.js` and composer views (`create.php`, `edit.php`) need full rewrites
- Lead form injection: replace `<!-- LEAD_FORM -->` placeholder in the HTML with the actual lead form markup before serving
