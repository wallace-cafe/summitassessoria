# ADR-002: Block Instance Storage as HTML Snapshot

## Status

Accepted

## Date

2026-05-20

## Context

When a landing page instance uses a block from the global library, the system must decide whether to store a live reference to the global template (block_template_id + token values only) or a self-contained snapshot of the HTML at save time plus token values. The choice has significant implications for data integrity of published pages and for operational complexity.

## Decision

Each block instance stored within a landing page's `blocks` JSON column will contain a **full snapshot** of the block's HTML template at the time the page is saved, along with the block's name, the original `block_template_id` (for informational purposes only), and the token values filled by the user.

The JSON structure for each block instance will be:
```json
{
  "id": "uuid-or-sequential-local-id",
  "block_template_id": 5,
  "block_name": "Card de Serviços",
  "html_snapshot": "<section class=\"services\">%TITULO%...</section>",
  "tokens": {
    "TITULO": "Consultoria Jurídica",
    "DESCRICAO": "Especializada em direito empresarial"
  },
  "position": 0
}
```

## Alternatives Considered

### Alternative: Live Reference (block_template_id + token values only)

- **Description**: Store only the `block_template_id` and token values. When rendering, fetch the current HTML from `block_templates` and substitute tokens.
- **Pros**: Smaller storage footprint; editing a global template automatically updates all pages that use it.
- **Cons**: Editing a global template unintentionally breaks or changes published live pages. Deleting a global template would require cascading logic or orphaned rendering errors. Creates a silent dependency between the block library and every published page.
- **Why rejected**: The uncontrolled propagation of changes to live pages is a high-impact production risk that outweighs the convenience. PRD explicitly flags this as a risk to mitigate.

## Consequences

### Positive

- Published pages are fully isolated from changes to the global block library.
- Deleting a global block template does not break any published page.
- The public renderer (`PublicController::show`) needs no JOIN to `block_templates` — it reads only `landing_pages.blocks`.
- Simplifies the public rendering pipeline significantly.

### Negative

- If the user edits a global block template to fix a design error, they must re-save every page that uses it to pull in the updated HTML. There is no automated propagation.
- Storage is slightly larger (full HTML per instance rather than just an ID reference).

### Risks

- Users may not realize that saving a page is required to pick up global template changes — the UI must communicate this clearly.

## Implementation Notes

- The `block_template_id` field in the snapshot is retained for Phase 3 tooling (e.g., detecting which pages use a given template, or offering a "refresh from template" action).
- Token substitution on save is NOT done at save time — the raw `%TOKEN%` HTML is stored in `html_snapshot`, and substitution happens at render time (both in the JS preview and in the PHP public renderer). This keeps the stored data flexible.
