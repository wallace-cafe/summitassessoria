# ADR-004: Database Schema — Replace `content` Column, Add `blocks` + `custom_css`

## Status

Accepted

## Date

2026-05-20

## Context

The existing `landing_pages` table has a `content TEXT NOT NULL` column used to store raw plaintext. The page builder requires two new storage fields: `blocks` (a JSON array of block instances) and `custom_css` (a CSS string). A decision is needed on whether to add these fields alongside `content` or replace it.

## Decision

Replace the `content` column with two new columns via an additive migration that:

1. Adds `blocks TEXT` (nullable) to store the JSON array of block instances.
2. Adds `custom_css TEXT` (nullable) to store the page-scoped CSS string.
3. Drops the `content` column (or in SQLite, recreates the table without it, since SQLite does not support `DROP COLUMN` in older versions).
4. Updates `LandingPageModel::$allowedFields` to `['title', 'slug', 'blocks', 'custom_css']`.
5. Updates `LandingPagesController::store()` and `update()` to accept and persist the new fields.
6. Updates `PublicController::show()` and the public view to render from `blocks` + `custom_css` instead of `content`.

Since the database is SQLite (writable/database.db) and the project is in active development with no production data at risk (confirmed by context), discarding the `content` column is acceptable.

## Alternatives Considered

### Alternative: Add New Columns, Keep `content` Intact

- **Description**: Add `blocks` and `custom_css` as nullable columns; leave `content` in place. The renderer reads `blocks` if present, otherwise falls back to `content`.
- **Pros**: Zero risk to existing data; the fallback allows gradual migration.
- **Cons**: Increases schema complexity with a dead column once all pages are rebuilt; the dual-path renderer adds conditional logic that becomes permanent technical debt; existing `content` data is plaintext and has no migration path to block JSON anyway.
- **Why rejected**: The project is in development with no production records at risk. Keeping `content` creates unnecessary long-term debt. Clean schema is preferable.

## Consequences

### Positive

- Clean, forward-looking schema with no legacy fallback paths.
- Renderer is simpler: always reads `blocks` + `custom_css`.
- `LandingPageModel` validation rules are unambiguous.

### Negative

- Any existing landing page records in development databases will lose their `content` data. Development databases must be reseeded or migrated manually after running the new migration.
- SQLite's `DROP COLUMN` syntax requires a table-recreation migration (more verbose than a simple `ALTER TABLE DROP COLUMN`).

### Risks

- If the project transitions to a real SQLite production database before this migration, existing content will be lost. Mitigation: run migration immediately; no production deployment exists yet.

## Implementation Notes

- SQLite `DROP COLUMN` is supported since SQLite 3.35.0 (released 2021). CI4's SQLite3 driver supports `ALTER TABLE ... DROP COLUMN` if the SQLite version is ≥ 3.35. Verify the server's SQLite version; if older, use the table-recreation approach in the migration.
- Migration filename: `2026-05-20-000000_AddBlocksToLandingPages.php`
- `blocks` column stores a JSON string; PHP code uses `json_encode`/`json_decode`. No JSON column type needed for SQLite (it stores as TEXT).
- `custom_css` column stores raw CSS string; no escaping beyond standard HTML escaping at render time.
