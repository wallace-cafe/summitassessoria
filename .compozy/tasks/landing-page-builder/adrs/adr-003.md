# ADR-003: Client-Side iframe Preview via srcdoc

## Status

Accepted

## Date

2026-05-20

## Context

The landing page composer requires a live preview pane that updates as the user edits. The design question is whether the preview HTML is assembled by JavaScript in the browser and injected into an iframe's `srcdoc` attribute, or assembled server-side via a dedicated preview endpoint.

## Decision

The live preview is assembled entirely in **JavaScript on the client**, then injected into the iframe using the `srcdoc` attribute. No server round-trip occurs during preview. The JS preview engine:

1. Reads the current Hero config (title, subtitle, background image URL).
2. Iterates the ordered block instances; for each, substitutes `%TOKEN%` patterns in `html_snapshot` with the current token input values.
3. Prepends the page's custom CSS in a `<style>` tag.
4. Wraps everything in a minimal `<!DOCTYPE html>` skeleton.
5. Assigns the resulting string to `iframe.srcdoc`.

This function is called on every relevant DOM change (CSS editor change, Hero field input, block token input, block reorder).

## Alternatives Considered

### Alternative: Server-Side Preview Endpoint

- **Description**: JavaScript POSTs the current editor state to `POST /landing-pages/preview`, the PHP controller assembles and returns the full HTML, and JavaScript sets `iframe.src` to a data URL or the response body.
- **Pros**: Preview HTML is produced by the same PHP renderer used for the public page, guaranteeing exact parity.
- **Cons**: Requires a network request on every keystroke (debounced but still adds latency); adds a new server route; requires CSRF token management for the preview request; increases server load proportional to typing speed.
- **Why rejected**: Adds unnecessary complexity and latency for a preview that can be produced identically in JavaScript, since the rendering logic (token substitution via regex + CSS injection) is trivial to duplicate in JS.

## Consequences

### Positive

- Zero latency: preview updates feel instantaneous.
- No extra server routes needed.
- No CSRF handling required for preview.
- The preview JS function also doubles as the source-of-truth for what the server will store (same token substitution logic).

### Negative

- The token substitution logic exists in two places: JavaScript (for preview) and PHP (for public rendering). These must be kept in sync.
- `srcdoc` attribute has a ~4MB practical limit in modern browsers; for typical landing pages this is not a constraint.

### Risks

- If the PHP renderer uses different substitution logic from the JS preview engine, visual output may differ. Mitigation: the substitution algorithm is simple (`str_replace` / `String.replace` with the same `%TOKEN%` regex) and is covered by tests in both environments.

## Implementation Notes

- The iframe must have the `sandbox` attribute with appropriate permissions (`allow-same-origin` is NOT included to prevent the page's JS from accessing the parent frame; use `allow-scripts` only if block HTML legitimately requires it).
- Debounce preview refresh to ~150ms after the last input event to avoid excessive re-renders during fast typing.
