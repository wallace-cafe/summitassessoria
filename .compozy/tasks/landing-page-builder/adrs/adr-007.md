# ADR-007: Landing Pages Database Schema — file_path Column

## Status

Accepted

## Date

2026-05-20

## Context

The old block-based system stores page content in `blocks` (JSON TEXT) and `custom_css` (TEXT) columns on the `landing_pages` table. The new file-upload system replaces these with files on disk. The table needs to be updated to reference the filesystem location while keeping existing metadata columns (title, slug, timestamps).

## Decision

Add a single `file_path` VARCHAR column to `landing_pages` storing the relative directory path from `writable/` (e.g., `landing_pages/meu-slug`). Remove the `blocks` and `custom_css` columns. The page's complete file set is always at `writable/{file_path}/`.

## Alternatives Considered

### Alternative 1: Individual File Path Columns

- **Description**: Add separate VARCHAR columns for each file type: `html_path`, `css_path`, `js_path`.
- **Pros**: Explicit per-file references, easy to locate a specific file without scanning the directory.
- **Cons**: Schema changes needed for every new file type added in the future; hard to represent multiple image files.
- **Why rejected**: Inflexible — cannot handle variable numbers of images without additional tables or JSON columns.

### Alternative 2: JSON `files` Column

- **Description**: Store a JSON object listing all uploaded files with their paths and types (e.g., `{"html":"index.html","images":["img1.jpg","img2.png"]}`).
- **Pros**: Flexible and self-describing, easy to extend.
- **Cons**: Over-engineered for MVP — the file_path directory contains everything needed; a directory scan is trivially fast.
- **Why rejected**: YAGNI. A filesystem directory IS the file manifest. Parsing JSON on every request to discover files adds unnecessary complexity when the filesystem already knows what exists.

## Consequences

### Positive

- Minimal schema change: one new column, two removed
- Filesystem is the single source of truth for file inventory — no sync between DB and disk
- Easy to locate a page's files: `writable/` + `file_path`
- Adding new file types in the future requires no schema migration

### Negative

- Serving files requires a directory read to discover available files (negligible cost for typical 3-15 file pages)
- Cannot query the database for pages that contain a specific file (not a use case for MVP)

### Risks

- **Orphaned file_path values**: If a row has a `file_path` but the directory is missing, the page is broken. Mitigation: the `show()` and `edit()` controllers check directory existence and log a warning if missing. The upload controller always creates the directory.
- **Path traversal in file_path**: The `file_path` value must be validated to only contain safe characters (`a-z`, `0-9`, `/`, `-`, `_`). Mitigation: validate on input and store sanitized value.

## Implementation Notes

- New migration: add `file_path` VARCHAR(255), drop `blocks` and `custom_css`.
- `LandingPageModel->allowedFields` updated to `['title', 'slug', 'file_path']`.
- On upload, `file_path` = `'landing_pages/' . $slug`.
- On slug change, the directory must be renamed to match (rare operation tracked as a risk).
