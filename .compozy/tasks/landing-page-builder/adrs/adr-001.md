# ADR-001: Block Library + Page Composer Architecture

## Status

Accepted

## Date

2026-05-20

## Context

The dashboard needs a landing page builder at `/landing-pages/create`. The core design question is how to structure the relationship between HTML block templates (with dynamic token placeholders) and the landing pages that use them. Three approaches were considered: a global block library with a per-page composer, inline blocks local to each page, or a gallery of pre-built templates. The user already expressed preference for writing raw HTML manually and for global block reusability across pages.

## Decision

Adopt the **Block Library + Page Composer** architecture:

1. A **global block library** (`/landing-pages/blocks`) where users create, name, and store reusable HTML block templates containing `%TOKEN%` placeholders.
2. A **per-page composer** (`/landing-pages/create` and `/landing-pages/edit/:id`) where users:
   - Edit a page-specific CSS stylesheet
   - Configure the fixed Hero block (title, subtitle, background image)
   - Pick blocks from the global library, fill in token values, and reorder them
3. All editor state is managed client-side in JavaScript and saved in a single server call.
4. A live preview renders the assembled page (with tokens substituted) alongside the editor.

## Alternatives Considered

### Alternative B: Inline Block Creation (Local Blocks)

- **Description**: Blocks are created and stored directly within each landing page. No global library.
- **Pros**: Simpler data model; fewer routes and screens.
- **Cons**: No reusability — the same block HTML must be recreated for every new landing page. Contradicts the user's explicit requirement for global reusability.
- **Why rejected**: Fails the reusability requirement stated during clarification.

### Alternative C: Template Gallery (Pre-Built Templates, No Raw HTML)

- **Description**: System ships pre-built HTML block templates (Hero, About, Services, etc.) that users clone and fill with token values. No raw HTML editing.
- **Pros**: Lower barrier for non-technical users; consistent design output.
- **Cons**: Severely restricts flexibility. The user explicitly chose raw HTML authoring with manual token syntax.
- **Why rejected**: Contradicts the user's preference for writing HTML manually and having full control over block structure.

## Consequences

### Positive

- Blocks created once can be reused across unlimited landing pages, saving time.
- Separation of concerns: block authoring (HTML + tokens) is distinct from page assembly (composition + CSS).
- Client-side editor with a single save call delivers a smooth, interactive UX without page reloads.
- The architecture is extensible: block categories, version history, or sharing between accounts can be added later.

### Negative

- Requires two distinct management surfaces: the block library and the page composer, increasing UI surface area.
- If a shared block is edited, all pages using it may be affected (unless instances are stored as snapshots — an open question).

### Risks

- Users unfamiliar with HTML may struggle with the raw editor — mitigated by token autocomplete hints and a live preview pane.
- Token naming conventions (`%TOKEN_NAME%`) must be enforced to avoid rendering errors on the public page.

## Implementation Notes

- The block library will need a new DB table (e.g., `block_templates`) with columns: `id`, `name`, `html_template`, `created_at`, `updated_at`.
- The landing page `content` column should be migrated from plain TEXT to a JSON structure storing the Hero config, ordered block instances (with block_template_id + token values), and custom CSS.
- The public renderer will deserialize the JSON, substitute tokens, and output raw HTML within the page's custom CSS scope.
