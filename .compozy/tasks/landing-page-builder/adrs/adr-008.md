# ADR-008: Block Editor HTML Parsing via DOMDocument

## Status

Accepted

## Date

2026-05-20

## Context

The block editor needs to parse `<!-- BLOCO_N_INICIO -->...<!-- BLOCO_N_FIM -->` comment delimiters from the uploaded `index.html`, extract the text content of each block for administrators to edit, then write changes back into the file. The HTML files are created by external designers and may contain complex markup, nested elements, and non-standard formatting.

## Decision

Use PHP's `DOMDocument` class (with `libxml` error suppression) to parse the HTML and find HTML comment nodes matching the `BLOCO_N_INICIO`/`BLOCO_N_FIM` pattern. For extraction, walk the sibling nodes between the two comment markers and collect only text nodes (`DOMText`). For rewriting, replace the content between the markers (preserving the markers themselves) with the new text.

## Alternatives Considered

### Alternative 1: Regex (preg_match)

- **Description**: Use `preg_match('/<!-- BLOCO_\d+_INICIO -->(.*?)<!-- BLOCO_\d+_FIM -->/s', ...)` to extract block content.
- **Pros**: Fast, concise (3-5 lines), no DOM overhead.
- **Cons**: Fails on nested comments, CDATA sections, or unusual whitespace. Cannot reliably distinguish text from markup inside blocks. Rewriting is fragile.
- **Why rejected**: The PRD requires extracting only TEXT content (not HTML). Regex cannot reliably separate text nodes from markup in complex HTML.

### Alternative 2: String Operations (strpos/substr)

- **Description**: Use `strpos` to find delimiter positions and `substr` to slice content between them.
- **Pros**: Simplest possible approach, no dependencies, very fast.
- **Cons**: Cannot distinguish text nodes from HTML tags; extracting "only text" would require manual tag stripping. Brittle if delimiters appear inside HTML attributes or script tags.
- **Why rejected**: Same limitation as regex — cannot reliably separate text content from markup in arbitrary HTML.

## Consequences

### Positive

- DOMDocument correctly handles arbitrary HTML structure
- Can distinguish text nodes from element nodes for accurate text-only extraction
- Comments are proper DOM nodes, making delimiter detection straightforward
- Can use `DOMXPath` for clean querying of block delimiters

### Negative

- `DOMDocument` is slower than regex for very large HTML files (negligible for typical landing pages under 100KB)
- Requires `libxml` error suppression (`@`) to handle non-standard HTML that designers may produce
- More verbose PHP code (~30 lines vs 5 lines for regex)

### Risks

- **Malformed HTML**: Designers may produce HTML with unclosed tags or other issues that cause `DOMDocument` to misinterpret the structure. Mitigation: use `libxml_use_internal_errors(true)` and `@$dom->loadHTML()` to suppress warnings; validate that expected comment nodes are found.
- **Very large files**: A landing page with hundreds of blocks could be slow to parse. Mitigation: blocks are expected at most 20-30 per page; performance is not a concern within this range.
- **Script/style content matching delimiters**: Designers should not put `BLOCO_N_INICIO` inside `<script>` or `<style>` tags. Mitigation: DOMDocument treats these as text nodes, but the editor only searches for comments at the document level — script/style content won't accidentally create phantom blocks.

## Implementation Notes

- `DOMDocument::loadHTML()` with UTF-8 encoding specified
- `DOMXPath::query('//comment()')` to find all comment nodes
- Filter comments matching `BLOCO_\d+_INICIO` and `BLOCO_\d+_FIM` patterns
- For extraction: collect all `DOMText` sibling nodes between the two comment markers
- For rewriting: rebuild the block content by replacing text between the marker comments, keeping the original markers intact
- Use `DOMDocument::saveHTML()` to write back the modified file
