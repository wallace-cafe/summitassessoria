# ADR-001: Dedicated API Controller with Bearer Token Filter

## Status

Accepted

## Date

2026-05-23

## Context

The dashboard-advogados project needs to expose landing page and leads data to an internal dashboard via a read API. The existing codebase uses session-based authentication for its web interface (AuthFilter) and already has ThrottleFilter for rate limiting. No external API surface exists yet. Two endpoints are required: one to list all leads for a given landing page (by slug) and one to list all registered landing pages. All requests must be authenticated using a static Bearer token stored in the `.env` file.

## Decision

Create a dedicated API controller (`App\Controllers\Api\LpController`) isolated from the existing web controllers, and a new `BearerTokenFilter` that reads a static token from the `.env` file (generated via CodeIgniter 4's native encryption facilities). The filter is applied to the `api/*` route group. Existing models (`LandingPageModel`, `LeadModel`) are reused without modification.

## Alternatives Considered

### Alternative 1: Add API methods to existing web controllers

- **Description**: Add JSON-returning methods directly to `LandingPagesController` and `LeadsController`
- **Pros**: Fewer new files; no structural changes
- **Cons**: Mixes session-based web logic with stateless API logic in the same class; the existing `AuthFilter` (session-based) conflicts with Bearer token routes; violates single-responsibility principle
- **Why rejected**: Coupling web and API concerns in the same controller creates maintenance risk and makes it harder to evolve either surface independently

### Alternative 2: Full API module (separate namespace/directory)

- **Description**: Create `app/Modules/Api/` with its own controllers, filters, routes, and config
- **Pros**: Maximum isolation; clean namespace for large API surfaces
- **Cons**: Significant structural overhead for only two endpoints; the project does not currently use CI4 modules, requiring a new architectural pattern
- **Why rejected**: YAGNI — the complexity is unjustified for an MVP with two endpoints

## Consequences

### Positive

- Clear separation between web (session) and API (Bearer token) authentication
- `BearerTokenFilter` is reusable for any future API endpoints added to the `api/*` group
- Existing models are reused with no changes — no regression risk
- Easy to understand and maintain for the internal dashboard team

### Negative

- Two new files added (controller + filter), slightly increasing codebase surface

### Risks

- Token exposure if `.env` is not properly secured on the server — mitigated by server-level `.env` access controls and the token never being committed to version control

## Implementation Notes

- Token should be generated using `php spark key:generate` or equivalent CI4 native method, stored as `API_TOKEN` in `.env`
- Filter reads token with `getenv('API_TOKEN')` and compares against the `Authorization: Bearer <token>` header
- All API responses follow the envelope pattern: `{ "data": [...], "meta": {}, "errors": null }`
