# ADR-002: Reuse `encryption.key` as the API Bearer Token

## Status

Accepted

## Date

2026-05-23

## Context

The API requires a static Bearer token to authenticate requests. The project already has a cryptographically strong key stored in `.env` under `encryption.key`, generated by `php spark key:generate` (which produces a 32-byte random key prefixed with `hex2bin:`). A dedicated separate token variable is an option, but introduces a second credential to manage. The PRD mandates use of CodeIgniter 4's native encryption facilities for token generation.

## Decision

The `BearerTokenFilter` reads the raw `encryption.key` value from the `.env` file using `getenv('encryption.key')` and compares it directly against the Bearer token string in the `Authorization` header. The client sends `Authorization: Bearer hex2bin:<hex-string>` using the exact value from `.env`. No separate `API_TOKEN` variable is created.

## Alternatives Considered

### Alternative 1: Dedicated `API_TOKEN` variable in `.env`

- **Description**: Generate a separate token string and store it as `API_TOKEN=hex2bin:...` in `.env`
- **Pros**: Allows rotating the API token independently of the encryption key; semantically clearer
- **Cons**: Two credentials to manage; the operator must run key generation twice; easy to forget to set `API_TOKEN` on first deploy
- **Why rejected**: Over-engineering for an MVP with a single internal consumer; `encryption.key` already exists and is guaranteed to be set by `php spark key:generate`

### Alternative 2: Generate a short random alphanumeric token

- **Description**: Use `bin2hex(random_bytes(32))` to produce a simple hex string stored in `.env`
- **Pros**: Token is shorter and easier to copy manually
- **Cons**: Does not use CI4's native encryption facility as required by the PRD technical constraint; introduces a custom generation step outside the framework
- **Why rejected**: Violates the PRD constraint that the token must use CI4's native encryption tooling

## Consequences

### Positive

- Zero additional configuration — `php spark key:generate` sets up both encryption and API auth in one step
- No new `.env` variable to document, forget, or misconfigure
- Token is cryptographically strong by default (256-bit, generated by `random_bytes`)

### Negative

- Rotating the encryption key also rotates the API token — the internal dashboard team must update their stored token whenever the key is rotated
- The token string (including the `hex2bin:` prefix) is long (~72 characters) — must be stored in a secrets manager rather than typed manually

### Risks

- Accidental key rotation via `php spark key:generate` invalidates API access — mitigated by documenting that `key:generate` should be run only once during initial setup

## Implementation Notes

- Filter reads: `$token = getenv('encryption.key');`
- Header comparison: `Authorization: Bearer <value-of-encryption.key-from-env>`
- The `hex2bin:` prefix is part of the token string — the client must include it verbatim
