# ADR-004: SQLite with WAL Mode via Migration and Password-Hashed Admin Seed

## Status

Accepted

## Date

2026-05-19

## Context

The PRD requires a self-contained, file-based database that works out of the box without external database server configuration. SQLite is the natural choice. The PRD also specifies WAL (Write-Ahead Logging) mode for optimized concurrent read/write performance, and mandates a seeded admin user with a default password ("123456"). We need to decide how to configure SQLite, enable WAL, and seed the initial user securely.

## Decision

1. **SQLite3 as the default database driver** configured in `app/Config/Database.php` with a file path in the `writable/` directory.
2. **WAL mode enabled inside a migration** via `PRAGMA journal_mode=WAL;` executed in the `up()` method of the initial migration. This ensures WAL is activated once when the database is created, not on every request.
3. **Password hashed with `password_hash()`** in the database seeder when creating the default admin user. The seed stores the bcrypt hash, not the plain text.
4. **Login verification uses `password_verify()`** in `AuthController::authenticate()`.

## Alternatives Considered

### Alternative 1: MySQLi with External Server

- **Description**: Configure MySQLi as the default driver and require a running MySQL/MariaDB server.
- **Pros**: Better concurrency for high-traffic scenarios; familiar to many PHP developers.
- **Cons**: Violates the PRD constraint of zero external database configuration; adds deployment complexity; WAL mode discussion becomes irrelevant.
- **Why rejected**: The PRD explicitly requires a file-based, self-contained database for out-of-the-box operation.

### Alternative 2: SQLite without WAL Mode

- **Description**: Use SQLite with the default rollback journal mode.
- **Pros**: Simpler; no need to run `PRAGMA` commands.
- **Cons**: Lower read concurrency; writers block readers; performance degrades with multiple simultaneous requests.
- **Why rejected**: The PRD specifically requests WAL mode for optimized read/write performance. The overhead of a single `PRAGMA` in migration is minimal.

### Alternative 3: Plain-Text Password in Seed

- **Description**: Store the admin password as "123456" directly in the database and compare with plain string in login.
- **Pros**: Simpler code; no hashing functions needed.
- **Cons**: Catastrophic security risk; violates every PHP security guideline; if database is exposed, credentials are exposed.
- **Why rejected**: Unacceptable security posture. `password_hash()` and `password_verify()` are built into PHP and require zero dependencies.

## Consequences

### Positive

- Zero external database dependency; works on any PHP host.
- WAL mode allows readers and writers to proceed concurrently, improving responsiveness under load.
- Password hashing protects the admin account even if the database file is compromised.
- All database setup is version-controlled (migrations + seeds) and reproducible via `spark migrate` and `spark db:seed`.

### Negative

- SQLite file must be writable by the web server process.
- WAL mode creates `-wal` and `-shm` sidecar files that must be backed up alongside the main database.
- SQLite is less suitable than client-server databases if the project later scales to multiple app servers.

### Risks

- WAL sidecar files (`*.db-wal`, `*.db-shm`) may be lost during careless file operations, corrupting the database. Mitigation: document backup procedures to include all three files.
- SQLite file locking can still be a bottleneck under extreme write concurrency. Mitigation: this is an admin dashboard with low write volume; if scale becomes an issue, migrate to PostgreSQL/MySQL later.

## Implementation Notes

- In `Database.php`, set the default connection to SQLite3 with `'database' => WRITEPATH . 'database.db'`.
- Create a migration (e.g., `2026-05-19-000000_CreateUsersTable`) that creates the `users` table and runs `$this->db->query('PRAGMA journal_mode=WAL;');`.
- Create a seeder (`AdminSeeder`) that inserts one user with `password_hash('123456', PASSWORD_DEFAULT)`.
- The `AuthController::authenticate()` method must use `password_verify($password, $user['password'])`.
- Run `php spark migrate` and `php spark db:seed AdminSeeder` as part of the initial setup.
