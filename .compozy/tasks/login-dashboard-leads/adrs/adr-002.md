# ADR-002: Separated Layout Architecture for Authentication and Dashboard

## Status

Accepted

## Date

2026-05-19

## Context

The admin interface requires two distinct visual contexts: a standalone login page and a full dashboard with sidebar navigation. The login page is a focused, minimal entry point, while the dashboard needs persistent navigation (sidebar) across all admin views. We need to decide whether to share a single layout with conditional logic or maintain dedicated layout files for each context.

## Decision

Create two separate layout files:
- `app/Views/layouts/auth.php` for the login page — minimal, no sidebar, centered form.
- `app/Views/layouts/dashboard.php` for all admin pages — includes sidebar with navigation links (Landing Pages, Manage Leads), top bar, and content area.

Each view extends its respective layout using CI4's `extend()` and `section()`/`renderSection()` mechanisms. CSS and JS specific to a view are injected via `renderSection('css')` and `renderSection('js')` defined in the individual view files.

## Alternatives Considered

### Alternative 1: Single Shared Layout with Conditional Logic

- **Description**: One `layouts/main.php` that conditionally renders the sidebar based on a variable passed from the controller.
- **Pros**: Fewer layout files; single source of truth for common HTML structure.
- **Cons**: Conditional logic in layouts reduces clarity; login page inherits unnecessary dashboard markup; harder to evolve independently.
- **Why rejected**: Mixing two distinct UX contexts in one file creates maintenance friction and violates separation of concerns. The login page should not carry dashboard-specific markup.

### Alternative 2: Layout per Page (No Layout Inheritance)

- **Description**: Each view is a self-contained HTML file with no `extend()` or layout system.
- **Pros**: Maximum independence; no learning curve for layout inheritance.
- **Cons**: Massive duplication of `<head>`, navigation, and footer markup across views; changes to global structure require touching every file.
- **Why rejected**: Unmaintainable for any project with more than two pages; contradicts DRY principles and CI4's view layout system.

## Consequences

### Positive

- Clear separation between public auth context and protected admin context.
- Dashboard layout can evolve (add breadcrumbs, notifications, user menu) without affecting the login page.
- Login page remains lightweight with no unused markup.
- CSS/JS sections allow per-view asset injection while keeping global assets in the layout.

### Negative

- Two layout files to maintain instead of one.
- Shared elements (meta tags, favicon, base CSS) must be kept in sync between `auth.php` and `dashboard.php`.

### Risks

- Shared head elements diverge over time. Mitigation: extract common head partials into `app/Views/partials/head.php` if duplication becomes problematic.

## Implementation Notes

- `auth.php` includes only the dark canvas background, centered card, and login form. No sidebar, no top nav.
- `dashboard.php` includes the full sidebar with links, content wrapper, and global `app.js` + `style.css` references.
- Use `$this->extend('layouts/dashboard')` in all admin views and `$this->extend('layouts/auth')` in the login view.
- Define `<?= $this->section('content') ?>` in both layouts and `<?= $this->endSection() ?>` in views.
- Add `<?= $this->section('css') ?>` / `<?= $this->section('js') ?>` in `dashboard.php` to allow per-view asset injection.
