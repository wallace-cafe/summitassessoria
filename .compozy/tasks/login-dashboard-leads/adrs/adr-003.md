# ADR-003: Resource-Oriented Controller Organization

## Status

Accepted

## Date

2026-05-19

## Context

The application requires handling authentication, a dashboard home page, landing page CRUD operations, and lead management. We need a controller organization strategy that balances clarity, maintainability, and alignment with CodeIgniter 4 conventions. The project is a clean CI4 starter with no existing controllers beyond `Home.php`.

## Decision

Adopt a resource-oriented controller organization with one controller per major domain:

- `AuthController` — `login()`, `logout()`, `authenticate()`
- `DashboardController` — `index()` (admin home)
- `LandingPagesController` — `index()`, `create()`, `store()`, `edit()`, `update()`, `delete()`
- `LeadsController` — `index()`, `filter()` (lead listing and source filtering)

Each controller extends `BaseController`, uses CI4's native session and validation services, and returns views or redirects. Public landing page rendering and lead capture belong to a separate `PublicController` to enforce the boundary between public and admin contexts.

## Alternatives Considered

### Alternative 1: Context-Grouped Controllers

- **Description**: `AuthController` for login/logout and `AdminController` that handles dashboard, landing pages, and leads in a single file.
- **Pros**: Fewer files; everything admin-related is in one place.
- **Cons**: `AdminController` becomes large and violates Single Responsibility Principle; harder to locate specific functionality; route-to-controller mapping becomes less intuitive.
- **Why rejected**: As the application grows, a monolithic admin controller becomes a maintenance bottleneck. Separating by resource keeps file sizes manageable and aligns with CI4 RESTful conventions.

### Alternative 2: Action-Based Controllers (Single Controller)

- **Description**: One `AppController` with methods for every action in the system.
- **Pros**: Single file to navigate.
- **Cons**: Extreme violation of SRP; impossible to scale; merge conflicts guaranteed in team settings.
- **Why rejected**: Antipattern for any non-trivial application. CI4's structure explicitly encourages controller separation.

## Consequences

### Positive

- Each controller has a single, well-defined responsibility.
- Route definitions map cleanly to controller classes (`/login` → `AuthController::login`).
- Future expansion (e.g., adding user profile, settings) only requires new controllers without touching existing ones.
- Aligns with CI4 documentation and community best practices.

### Negative

- More controller files than a monolithic approach.
- Slightly more boilerplate (each controller repeats `extends BaseController`, `initController` logic).

### Risks

- Over-fragmentation if taken to an extreme (one controller per action). Mitigation: group closely related actions in the same controller (e.g., all landing page CRUD stays in `LandingPagesController`).

## Implementation Notes

- `BaseController` will be updated to preload `session`, `form`, and `url` helpers for all controllers.
- `AuthController` and `PublicController` do not require auth filters; all others do.
- `LandingPagesController` uses CI4's validation rules inline for create and update forms.
- `LeadsController` handles search, sort, and landing-page source filter via query parameters.
