# ADR-001: Unified Asset Pipeline for Landing Pages

## Status

Accepted

## Date

2026-05-22

## Context

The current landing page creation route `/landing-pages/create` allows users to upload project images, which are validated and saved inside a nested `images/` subfolder (`writable/landing_pages/{slug}/images/`). 

With the growing need for rich media, we must expand this capability to allow both image and video uploads. Separating images and videos into isolated folders adds unnecessary complexity to the page-rendering engine, and the user needs a simple, unified asset directory.

## Decision

We will implement a **Unified Asset Pipeline** (Approach 1):
1. Rename the "Images" file input field to "Assets" on the `landing-pages/create` view, supporting both images (`.jpg, .png, .webp, .svg, .gif`) and videos (`.mp4, .webm, .ogg`).
2. Update the backend controller validation to support these file extensions and raise the upload size limit to 50MB.
3. Consolidate file saving under a single `assets/` folder in the landing page's slug directory (`writable/landing_pages/{slug}/assets/`).
4. Update the dynamic asset serving routing and logic in `PublicController` to fetch and render resources directly from the unified `assets/` directory with correct MIME types for both images and videos.

## Alternatives Considered

### Alternative 1: Dual Folder Isolation

- **Description**: Separate uploaded files based on extension, storing images in `images/` and videos in `videos/`.
- **Pros**: Keeps file directories separated by content type.
- **Cons**: Adds complexity to asset routing and page construction, requiring separate handling paths.
- **Why rejected**: Violated the user's explicit preference to keep all assets inside a single consolidated `assets/` folder.

## Consequences

### Positive

- **Simpler Architecture**: Reduces folder hierarchy, making path resolution and dynamically served assets easier to locate.
- **Support for Video**: Enables marketers to upload high-fidelity video formats directly.
- **Consolidated Handling**: Standardizes pathing for future optimizations (like CDN distribution).

### Negative

- **No Auto-Migration**: Previously created pages will not have their folders renamed automatically. The user has acknowledged and accepted that any older pages needing this layout must be manually recreated.

## Implementation Notes

- Form field renamed from `images[]` to `assets[]`.
- Controller updates validation to check `ext_in[assets,jpg,jpeg,png,webp,svg,gif,mp4,webm,ogg]` and `max_size[assets,51200]`.
- Destination folder changed from `$dir . '/images'` to `$dir . '/assets'`.
