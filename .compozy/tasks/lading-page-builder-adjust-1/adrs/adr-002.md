# ADR-002: Dynamic Flat Asset Serving and Validation Strategy

## Status

Accepted

## Date

2026-05-22

## Context

To implement the unified asset pipeline, we need to decide:
1. How routes will map to the new `assets/` directory.
2. How the controller will validate the uploaded image and video assets.
3. How `PublicController` will set correct MIME types for HTML5 video playback.

Since all uploaded files are stored flat within a single level under `assets/`, we must align the routes and controller logic around this layout.

## Decision

We will implement the following technical designs:
1. **Flat Route Resolution**: Continue utilizing the existing `p/(:any)/(:any)/(:any)` route, which parses requests for `/p/{slug}/assets/{filename.ext}` seamlessly using dynamic parameters.
2. **Standard CodeIgniter File Validation**: Apply native CodeIgniter framework validation rules for the `assets` upload field:
   - Size limit: 50MB (`max_size[assets,51200]`).
   - Allowed extensions: `.jpg, .jpeg, .png, .webp, .svg, .gif, .mp4, .webm, .ogg`.
3. **Explicit MIME Map with Fallback**: Extend the `$mimeMap` array inside `PublicController::asset` with explicit mappings for:
   - `mp4` => `video/mp4`
   - `webm` => `video/webm`
   - `ogg` => `video/ogg`
   And keep `mime_content_type()` as a generic fallback for any other cases.

## Alternatives Considered

### Alternative 1: Dynamic Multi-Segment Wildcard Route

- **Description**: Replace specific routes with a generalized `p/(:any)/(.+)` wildcard route.
- **Why rejected**: Unnecessary since the user specified that files will be stored in a flat structure inside `assets/` with no nested subfolders. The existing `p/(:any)/(:any)/(:any)` route handles 3-segment requests perfectly.

### Alternative 2: Custom Validation Loop

- **Description**: Validate files manually in a loop instead of utilizing standard CodeIgniter validation.
- **Why rejected**: Standard CodeIgniter rules are cleaner, built-in, and more secure than custom looping logic.

## Consequences

### Positive

- **Simplicity**: No routing changes are required since the existing dynamic routes already match flat `assets/` files perfectly.
- **Robust Video Rendering**: Hardcoded MIME types prevent browsers from failing to decode standard HTML5 videos.
- **Security**: The framework's native `uploaded` and `ext_in` rules provide standard server-side validation.

### Negative

- **No Nested Directory Support**: Subdirectories within `assets/` are not supported. This is fully aligned with user requirements.
